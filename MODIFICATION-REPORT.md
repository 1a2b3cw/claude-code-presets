## 一、核心痛点

从订阅管理项目的实际经验和审计结果，问题归结为 3 个：

| 痛点 | 根因 | 现状 |
|------|------|------|
| **需求不明确，写出来不是想要的** | 没有可行性确认环节，AI 不问就直接做 | dev.md 跳过确认直接开发 |
| **写完一个功能不知道质量如何** | 只有"全面审查"和"什么都不查"两档，中间态缺失 | review-all.md 太粗，没有功能级快检 |
| **出了问题不知道怎么修** | 流程只有正常路径，没有异常路径 | dev.md 没写"测试失败怎么办"、"审查打回怎么办" |

## 二、最完美开发流程设计

### 你说一句话，AI 自动走完全程

```
你说：/dev 做一个用户登录功能
          ↓
┌─ Phase 0: 需求确认（1 分钟）────────────────────────┐
│  AI 问你 3 个问题：                                    │
│  1. 这个功能解决什么问题？                              │
│  2. 有没有技术限制（第三方服务、现有接口）？              │
│  3. 做到什么程度算完成？（验收标准）                     │
│                                                        │
│  你回答后 → AI 输出一句话摘要 → 你说"确认"              │
│  如果你说"不对"→ AI 重新问，最多 3 轮                   │
│  3 轮还不清楚 → AI 说"我需要更多信息"，暂停等待          │
└──────────────────────────────────────────────────────┘
          ↓
┌─ Phase 1: 复杂度判断 + 规划（自动）──────────────────┐
│  S 级（1-10 行）→ 跳到 Phase 2，不规划                │
│  M 级（单模块） → 输出 5 行 checklist，你确认           │
│  L/XL 级       → 输出 spec.md + tasks.md，你确认       │
└──────────────────────────────────────────────────────┘
          ↓
┌─ Phase 2: 迭代开发（核心循环）───────────────────────┐
│                                                        │
│  for 每个任务 in tasks.md:                             │
│    ┌─────────────────────────────────────────┐         │
│    │ Step 1: Builder 写代码（TDD）            │         │
│    │ Step 2: /check 自动快检 ← 关键新增       │         │
│    │   → 逻辑问题？类型错误？边界遗漏？        │         │
│    │   → 有问题：自动修 → 重新检查             │         │
│    │   → 没问题：继续                          │         │
│    │ Step 3: git commit                       │         │
│    └─────────────────────────────────────────┘         │
│                                                        │
│  所有任务完成后 → /review-all 全面审查                  │
│  审查不通过 → 自动修 → 重新审查（最多 2 轮）            │
│  2 轮不过 → 列出问题，等你决定                          │
└──────────────────────────────────────────────────────┘
          ↓
┌─ Phase 3: 验收（自动）───────────────────────────────┐
│  运行所有测试                                          │
│  类型检查 tsc --noEmit                                │
│  检查 git 状态（未提交的文件）                          │
│  输出结果表格：通过/未通过/需确认                        │
│  未通过 → 自动修 → 重新验收                             │
│  全部通过 → 输出"完成"                                  │
└──────────────────────────────────────────────────────┘
```

### 关键：颗粒度对齐

| 粒度 | 指令 | AI 行为 |
|------|------|---------|
| **粗** | `/dev 做一个用户登录` | 走完全流程，你只管确认 |
| **中** | `/check` | 快检当前代码，1 分钟出结果 |
| **细** | `/fix 这个函数返回值类型不对` | 直接改这一处，不走流程 |
| **查错** | `/check src/auth/login.ts` | 快检这一个文件 |
| **看进度** | `/standup` | 告诉你做到哪了 |

### 最关键的三个问题

1. **security-check.sh 始终返回 exit 0**：安全 hook 发现 eval()、innerHTML 等严重安全问题时只打印警告不阻止操作。与 "安全检查贯穿全程" 的设计目标直接矛盾。应该对"严重"级别返回 exit 2。

2. **dev.md 缺少异常路径**：测试失败、用户否决 spec、审查打回 -- 这些都是常见场景，但流程没有覆盖。Claude 遇到这些情况只能靠自己判断。

3. **review-all.md 范围未定义**：不带参数时审查什么不清楚，审查完之后的后续动作也没有。Claude 使用这个命令时行为不一致。

### CLAUDE.md 详细问题

- 缺少错误处理指导：流程图描述了正常路径，没有异常路径
- "迭代可中断：发现方向错了，随时回退调整"说了但没说怎么退
- Phase 0 Spike 没有退出标准：什么情况下判定"不可行"不清楚
- "用户确认后进入迭代"：如果用户不确认呢？循环几轮不清楚
- M 级 tasks.md 格式没有定义，只说"轻量 checklist"无示例

### dev.md 详细问题

- Step 4 验收太薄：测试失败、集成验证发现新 bug 没有回退路径
- 没有说谁执行每个 Step，措辞模糊
- S/M 级跳到 Step 3 但 Step 3 写的是"对 tasks.md 中的任务分批迭代"，S 级没有 tasks.md
- 没有提到 git 操作：什么时候 commit、什么时候 branch

### review-all.md 详细问题

- 内容只有 23 行，几乎所有内容都指向 code-review SKILL.md
- 没有说审查范围怎么确定（不带参数时审查什么）
- 没有输出后续动作（审查完了然后呢）

### security-check.sh 详细问题

- 始终返回 exit 0（第 66 行），即使发现严重问题也只打印警告不阻止
- SQL 拼接检测太弱：只检测 SELECT...WHERE...+ 模式
- 跳过 .sh 文件，shell 脚本中的安全问题不会被检查
- 硬编码密钥检测只匹配 8+ 字符

### devops.md 详细问题

- 内容只有 35 行，没有工作流程、没有输出格式、没有具体步骤
- 所有实质内容都指向 ci-cd-pipelines SKILL，自身几乎没有独立内容
- 没有说什么时候介入（Phase 2 还是 Phase 3）

### Rules 和 Skills 重叠

- project-planning SKILL 与 Architect-Planner Agent 高度重叠
- testing SKILL 与 testing rules 高度重叠
- react rules 与 frontend SKILL 内容重叠
- typescript rules 与 typescript-advanced SKILL 关系不明
