## 第一次使用：5 分钟上手

### 启动前检查

```bash
# 1. 确认在项目目录
cd D:\CD\MY1

# 2. 启动 Claude Code
claude

# 3. 验证 MCP 服务器（输入后应该看到 5 个服务器）
/mcp
```

### 第一个命令

进入 Claude Code 后，试试这个：

```
/dev 做一个简单的 TODO 应用，用 Next.js + TypeScript
```

Claude 会自动判断这是 L 级任务，执行完整流程：规划 → 迭代开发 → 审查 → 完成。

**你只需要做两件事**：提需求、确认方案。代码、测试、审查全部自动完成。

## 核心指令速查表

项目有 6 个斜杠命令，用错命令不会出错，但用对命令效率更高：

| 命令 | 什么时候用 | 一句话说明 | 类比 |
|------|-----------|-----------|------|
| `/dev` | **要写代码的时候** | 从需求到代码的完整流程 | "帮我做个东西" |
| `/check` | **写完一个功能想快检** | 3 维度快检（逻辑+类型+边界），1 分钟 | "帮我快速看看" |
| `/fix` | **修一个已知的具体问题** | 指定文件或描述问题，直接修 | "这里有个 bug" |
| `/review-all` | **代码写完了想全面检查** | 跨文件审查（变更完整性+一致性+历史回归+依赖关系） | "帮我检查一下" |
| `/ship` | **准备发布上线** | 发布前最终检查清单 | "能上线了吗" |
| `/standup` | **想知道进度** | 生成项目状态汇报 | "现在啥情况" |

### 记忆技巧

```
开发用 /dev → 快检用 /check → 修 bug 用 /fix → 审查用 /review-all → 上线用 /ship → 看进度用 /standup
```

## 场景实战

### 场景 1：做一个新功能

**你说**：
```
/dev 给用户个人页面添加头像上传功能
```

**会发生什么**：
1. Claude 判断这是 L 级任务（跨模块：前端组件 + 后端 API + 文件存储）
2. Architect-Planner 规划：输出 spec.md（做什么）+ tasks.md（怎么做）
3. Claude 问你："方案是这样的，确认吗？"
4. 你说"确认"，进入迭代开发
5. Builder 写代码 + Reviewer 自动审查，发现问题自动修
6. 全部完成后告诉你"做完了"

**你需要做的**：
- 确认方案（看一眼合不合理）
- 有疑问随时打断："这个方案不对，应该用 OSS 存储而不是本地"

**为什么这么设计**：
> 初学者常犯的错误是直接说"帮我写代码"，跳过规划。结果写出来的东西和预期不一样。
> `/dev` 强制先规划再编码，确保你和 Claude 对"要做什么"达成一致。

---

### 场景 2：修一个小 Bug

**你说**：
```
/dev 修复登录页面输入空密码时没有提示的问题
```

**会发生什么**：
1. Claude 判断这是 S 级（小修复）或 M 级（需要加验证逻辑）
2. S 级：直接修，不规划
3. M 级：快速规划 → 修 → 审查

**为什么不用 `/review-all`**：
> `/review-all` 是"检查已有代码"，不是"修代码"。修 Bug 用 `/dev`，修完 Claude 会自动审查。

---

### 场景 3：检查已有代码

**你说**：
```
/review-all src/features/auth/
```

**会发生什么**：
/review-all 聚焦**跨文件关系**，从 4 个维度分析：
1. 变更完整性（改了接口，所有引用方都同步了吗？）
2. 跨文件一致性（同类功能在不同文件中用了相同的模式吗？）
3. 历史回归（之前被打回的问题这次修了吗？）
4. 依赖关系（新增/删除了依赖，配置都同步了吗？）

单文件质量由 code-review skill 负责（6 维度），/review-all 不重复做。输出一份跨文件审查报告，有问题自动修。

**什么时候用**：
- 写完一批代码想检查质量
- 合并 PR 前想审查
- 学习别人的代码想了解问题

**为什么不直接用 `/dev`**：
> `/dev` 的审查是自动的、内嵌在开发流程中的。`/review-all` 是你主动发起的、独立的审查，可以审查任何代码（包括不是 Claude 写的）。

---

### 场景 4：准备上线

**你说**：
```
/ship
```

**会发生什么**：
Claude 按清单逐项检查：
- 测试全过了吗？
- 类型检查过了吗？
- 有安全漏洞吗？
- 环境变量配好了吗？
- 数据库迁移准备好了吗？
- 无障碍达标了吗？

输出一个表格，告诉你哪些过了、哪些没过、怎么修。

**什么时候用**：
- 觉得功能做完了，想发布
- 跑 CI/CD 之前想先本地检查
- 给领导/客户演示之前

**为什么需要这个**：
> 人总会忘记东西。`/ship` 是一个标准化的检查清单，确保你不会带着 Bug 或安全漏洞上线。

---

### 场景 5：了解项目状态

**你说**：
```
/standup
```

**会发生什么**：
输出一份简报：
- 哪些任务完成了
- 哪些在进行中
- 有没有阻塞项
- 下一步计划

**什么时候用**：
- 早上开工想了解昨天做了什么
- 想给领导汇报进度
- 中断了工作想快速恢复上下文

---

### 场景 6：不知道该用什么技术

**你说**：
```
这个项目的状态管理用 Zustand 还是 Redux Toolkit？帮我分析一下
```

**会发生什么**：
1. Researcher 用 Context7 查两个库的最新文档
2. 对比分析：优缺点、适用场景、社区活跃度
3. Architect-Planner 结合你的项目需求给出建议
4. 输出 research.md

**为什么不直接问 Claude**：
> Claude 的知识有截止日期，但 Context7 能查到最新的库文档。用 MCP 查文档比靠记忆回答更准确。

---

### 场景 7：想看看页面效果

**你说**：
```
打开 http://localhost:3000 看看首页效果，截图给我
```

**会发生什么**：
Playwright MCP 启动浏览器，打开页面，截图返回给你。

**更高级的用法**：
```
打开 http://localhost:3000，点击登录按钮，输入用户名 test 和密码 123456，
验证是否跳转到了首页
```
这会自动执行一个完整的 E2E 测试流程。

## MCP 工具：什么时候用什么

MCP 工具是 Claude 的"外挂能力"，让 Claude 能操作外部系统。

### GitHub MCP

**什么时候用**：
```
帮我查看这个仓库最近的 PR
创建一个 Issue：标题"修复首页样式"，标签 bug
在仓库中搜索所有包含 "useState" 的文件
```

**为什么需要**：
> 不用离开 Claude Code 就能管理 GitHub，不用切浏览器。

### Context7 MCP

**什么时候用**：
```
查一下 Next.js 14 的 App Router 怎么用
React 18 的 useDeferredValue 怎么用？
Hono 框架的中间件怎么写？
```

**为什么需要**：
> Claude 的训练数据有截止日期。Context7 能查到最新的库文档，避免给你过时的写法。

### Playwright MCP

**什么时候用**：
```
打开 http://localhost:3000 并截图
帮我写一个 E2E 测试：用户注册流程
点击页面上的提交按钮，看看有没有报错
```

**为什么需要**：
> 代码写完了不等于能用。Playwright 让 Claude 能实际操作浏览器，验证功能是否正常。

### SQLite MCP

**什么时候用**：
```
创建一个 users 表，包含 id、name、email、created_at
往 users 表插入一条测试数据
查询所有邮箱包含 @example.com 的用户
```

**为什么需要**：
> 开发阶段用 SQLite，不用装 PostgreSQL。Claude 可以直接操作数据库，不用你写 SQL。

### PostgreSQL MCP

**什么时候用**：
```
显示 users 表的结构
查询最近 7 天注册的用户数量
分析这条 SQL 的执行计划
```

**为什么需要**：
> 生产环境用 PostgreSQL。Claude 可以帮你写查询、分析性能、检查表结构。

## 直接调用 Agent 和 Skill

大多数时候用 `/dev` 就够了。但有些场景直接调用更高效：

### 什么时候直接调用 Agent

| 场景 | 怎么说 | 为什么 |
|------|--------|--------|
| 只想规划不想写代码 | "请 Architect-Planner 帮我规划一下这个功能" | `/dev` 会直接开始写代码，但你可能只想先看看方案 |
| 只想审查特定文件 | "请 Reviewer 审查一下 src/auth/ 目录" | 比 `/review-all` 更精确，可以指定审查范围 |
| 想调研技术方案 | "请 Researcher 调研一下 Prisma 和 Drizzle 的区别" | 只需要调研，不需要写代码 |

### 什么时候直接调用 Skill

| 场景 | 怎么说 | 为什么 |
|------|--------|--------|
| 想用 TDD 方式开发 | "按照 testing skill 的 TDD 流程来" | 强制先写测试再写代码 |
| 想优化性能 | "按照 performance skill 检查一下这个页面" | 专注于性能维度 |
| 想设计 API | "按照 api-design skill 设计用户管理 API" | 遵循 RESTful 规范 |

## 进阶技巧

### 1. 任务分级意识

```
# S 级（直接说，不需要 /dev）
把首页标题改成"欢迎回来"

# M 级（用 /dev，但可以跳过规划）
/dev 给用户列表加一个搜索框

# L 级（用 /dev，完整流程）
/dev 做一个用户认证系统

# XL 级（用 /dev，可能需要多轮）
/dev 重构整个支付模块，支持多种支付方式
```

**为什么要区分**：
> 小任务走完整流程是浪费时间。大任务跳过规划是冒险。`/dev` 会自动判断，但你主动说明级别可以让 Claude 更快响应。

### 2. 给 Claude 上下文

```
# 不好：太模糊
/dev 做一个管理后台

# 好：具体明确
/dev 做一个管理后台，包含用户管理、订单列表、数据统计三个页面。
技术栈用 Next.js + Prisma + PostgreSQL。
用户管理需要 CRUD，订单列表需要分页和筛选，数据统计需要图表展示。
```

**为什么**：
> Claude 不知道你的业务背景。给的信息越多，规划越准确，返工越少。

### 3. 随时打断和调整

```
# 开发过程中发现问题
等等，这个方案不对。用户头像应该存在 OSS 上，不是本地。

# 觉得方向错了
停一下，我重新想了一下，不需要这么复杂，用 session 就行，不用 JWT。

# 想加功能
对了，登录还要加一个"记住我"功能。
```

**为什么可以这样做**：
> Claude 的迭代模型支持随时调整。发现问题越早，修改成本越低。

### 4. 让 Claude 解释代码

```
帮我解释一下 src/features/auth/services/auth-service.ts 里的逻辑
这段代码为什么要用 try-catch 包起来？
这个中间件是做什么的？
```

**为什么有用**：
> 初学者最大的问题是"看不懂代码"。让 Claude 解释比自己啃快 10 倍。

### 5. 让 Claude 写测试

```
给 src/utils/validate-email.ts 写单元测试
帮我写一个 E2E 测试：用户注册 → 登录 → 查看个人页面
```

**为什么单独提**：
> `/dev` 会自动写测试，但有时候你想给已有代码补测试，或者想用特定的测试方式（比如 E2E）。

### 6. 用 Context7 查文档

```
# 不好：靠 Claude 的记忆
React 的 useCallback 怎么用？

# 好：让 Claude 查最新文档
用 Context7 查一下 React 18 的 useCallback 最新用法和最佳实践
```

**为什么**：
> React 19 的行为可能和 React 18 不一样。查文档比靠记忆靠谱。

## 常见错误和避坑

### 错误 1：一句话需求

```
# 不好
/dev 做个网站

# 为什么不好：Claude 不知道你要什么网站，会按最简单的方式做，
# 结果大概率不是你想要的。

# 好
/dev 做一个个人博客网站，用 Next.js + MDX，支持暗色模式，
首页展示文章列表，点击进入文章详情页。
```

### 错误 2：跳过确认

```
# 不好：Claude 给了方案，你看都不看就说"确认"
# 结果可能和你想的不一样

# 好：花 30 秒看一眼方案
# - 功能范围对不对？
# - 技术选型合不合理？
# - 有没有遗漏的需求？
```

### 错误 3：不给反馈

```
# 不好：Claude 做完了，你觉得不对但不说
# Claude 以为你满意，下次还这么做

# 好：直接说
这个样式不对，按钮应该在右边。
这个 API 的响应格式改成 { code, data, message }。
```

### 错误 4：滥用 /review-all

```
# 不好：每写一行代码就跑一次 /review-all
# 浪费时间，而且代码还没写完

# 好：写完一个完整功能后再审查
/review-all src/features/auth/
```

### 错误 5：忽略安全检查

```
# 不好：安全检查 Hook 报了警告，直接忽略继续
# 可能导致线上安全漏洞

# 好：认真看警告内容
# - 如果是误报（比如注释里的示例代码），可以忽略
# - 如果是真问题（比如 SQL 拼接），必须修
```

### 错误 6：不写测试

```
# 不好：觉得写测试浪费时间，让 Claude 跳过
# 结果上线后出 Bug，修 Bug 的时间远超写测试的时间

# 好：让 Claude 按 TDD 流程来
# 先写测试 → 测试失败 → 写代码 → 测试通过
# 这样你对代码有信心，改代码也不怕
```

## 一句话总结

| 你的状态 | 该做什么 |
|----------|----------|
| 想做新功能 | `/dev` + 具体需求描述 |
| 代码写完了 | `/review-all` 检查一下 |
| 要上线了 | `/ship` 最终检查 |
| 想看进度 | `/standup` |
| 不确定用什么技术 | 直接问，Claude 会用 Context7 查文档 |
| 想看页面效果 | 直接说"打开 XXX 截图" |
| 看不懂代码 | 直接说"帮我解释这段代码" |
| 遇到 Bug | `/dev 修复 XXX 问题` |

**核心原则**：你负责想清楚"要什么"，Claude 负责"怎么做"。
