# 调试方法

> 通用方法论，不绑定语言。具体调试工具按你的栈选（前端 Chrome DevTools、Node Inspector、Python pdb/debugpy 等）。

## 触发条件
- 遇到 bug / 测试失败 / 行为不符预期 / 性能异常

## 调试流程

### 1. 复现
- 确认能**稳定复现**（偶现的先想办法稳定）
- 记录复现步骤和影响范围

### 2. 缩小范围
- **二分法**：禁用一半，看问题还在不在，逐步逼近
- **日志法**：在关键路径打印输入/输出/状态
- **断点法**：用 debugger 单步执行
- **对比法**：和能正常工作的版本/分支对比

### 3. 定位根因
问自己：
- 最近改了什么？什么时候开始坏的？（`git bisect` 是利器）
- 有没有错误信息/堆栈？看**最底层**的那个原因
- 是代码问题，还是环境/数据/依赖问题？
- **不要假设，要验证**——每个怀疑点都用证据确认

### 4. 修复
1. 先写一个**复现 bug 的失败测试**
2. 修代码让测试通过
3. 检查是否引入新问题（看相关代码）

### 5. 验证
- 跑完整测试套件
- 手动验证修复 + 检查相关功能没被带坏

## 常见根因分类（通用）

| 类别 | 常见原因 |
|------|----------|
| 逻辑 | 边界未处理、条件不全、off-by-one、空值 |
| 异步/并发 | 竞态、未 await、未处理 reject、顺序依赖 |
| 数据 | 脏数据、类型不符、编码、时区 |
| 集成 | 接口契约不一致、版本不兼容、超时 |
| 环境 | 依赖版本、配置/环境变量、权限、网络 |

## 调试原则
- 先理解，再修复（别瞎试）
- 一次只改一处
- 修完写测试防回归
- 不假设，用证据验证
- 记录根因和解决方案（写入 `workspace/journal.md`）
