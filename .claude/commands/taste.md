# /taste - 设计方向探索

不知道想要什么 UI 风格？这个命令带你**找到**它，并把它**固化成项目的设计方向**。**只定方向，不写代码。**

## 定位：补上"设计方向从哪来"

体系规定"涉及 UI 必须先声明设计方向"（写进 `preview/` 或 spec），却没说这个方向怎么定——全靠拍脑袋，于是常常做完才发现"丑/死板"再返工。`/taste` 就是那个缺失的过程。

| | /plan | /taste | /dev |
|---|-------|--------|------|
| **产出** | `roadmap.md` | `preview/` 设计方向 | 可运行代码 |
| **何时用** | 项目开局 | 写 UI 前、没想好风格 | 已有方向，开造 |
| **写代码吗** | ❌ | ❌ | ✅ |

典型流程：`/plan 我想做个 X` → `/taste`（涉及 UI 时）→ `/dev 做模块 1`。

## 何时用
- 新项目要做前端，但说不清想要什么风格
- 觉得现有界面"丑 / 死板"，想重新定调
- **跳过**：项目已有 `preview/` 设计方向，或风格已经很明确

## 核心理念：找审美分三层

| 层 | 是什么 | 频率 | 存哪 |
|----|--------|------|------|
| **L0 个人审美档案** | 跨项目的你的偏好（明暗/繁简/色系/忌讳） | 一次建档，偶尔更新 | `~/.claude/taste-profile.md`（可选） |
| **L1 项目设计方向** | 结合本项目行业/用户，定这个项目长什么样 | 每项目一次 | `preview/`（本命令核心产出） |
| **L2 落地微调** | 按方向写代码 + 截图回看调细节 | 每次写 UI | 由 /dev 完成 |

**L0 一次建档、长期复用；每个新项目只做 L1。** 有了 L0，新项目定向会越来越快。

## 流程

### 入口：选一条路（随时可换）

不强迫用户先有设计词汇。直接问：

> **"想从哪条路开始找你的审美？选了之后我就按这条往下带，随时可换。"**

- **A 生成情绪板** — 用图像生成 MCP 现场出 4-6 张差异**很大**的界面风格图，用户不用想、凭直觉挑顺眼的（最直观）
- **B 快问快答** — 一组组明显对立的方向二选一（冷 vs 暖、留白 vs 饱满、明 vs 暗、摄影 vs 插画…），从用户的反应反推偏好画像（最省脑、不依赖已有词汇）
- **C 逛真实参考** — 用 WebSearch 找高水准的真实网站/后台/作品集推给用户，用户去看，把心动的截图丢回来（最养眼、攒具体例子）
- **D 组合** — 先 B 圈大方向，再 A 细化

### 收敛
用户选中 → 同方向生成更多变体细化，或换方向重来。可多轮，**路可随时换**，不背沉没成本。每一轮都把确定下来的偏好记下来，逐步拼出画像。

> **自检闭环**：生成或选定后，用 Playwright/读图把结果**截图回看一遍再给用户**——别盲给。看到了才好收敛。

### 沉淀（真正的产出，不止是几张图）
方向定了，必须落成**可执行**的设计方向，否则又回到拍脑袋：
- **必出**：项目根 `preview/design-direction.md`（见下方格式）
- 可选：把选中的情绪板图存进 `preview/`
- 可选：更新 `~/.claude/taste-profile.md`（个人档案 +1，下个项目更快）

### 交棒
> "设计方向已写入 `preview/`。说 `/dev 做 X` 开始，Builder 会按 `preview/` 实现，Designer 审查视觉。"

## 工具依赖与降级
- **A 情绪板** 需要图像生成 MCP（如通义万相 `dashscope-image` 等任意文生图服务）。**没有就自动降级到 B/C**，定向照样能完成。
- **C 参考库** 用内置 WebSearch / WebFetch，无需额外配置。
- **L2 截图回看** 用 `playwright` MCP（底座自带）。

不强制任何额外 MCP——缺图像 MCP 时，B（快问快答）+ C（逛参考）一样能把方向定下来。

## `preview/design-direction.md` 输出格式

```markdown
# [项目名] 设计方向
> 由 /taste 于 YYYY-MM-DD 探索得出

## 风格
- 风格名：[如"温暖品牌" / "暗色高级"]
- 一句话气质：[沉稳科技 / 亲和温暖 / …]
- 目标用户：[谁在用，年龄/场景]

## 色板
- 背景 / 表面 / 主文字 / 次文字：[hex …]
- 主色（强调，约 10%）：[hex] ｜ soft 底：[hex]
- 语义：成功 / 警告 / 危险 [hex …]

## 形与字
- 字体：中文栈 + 拉丁栈；字号基调（正文 ?px / 辅助 ?px）
- 圆角：卡片 ? / 按钮 ? / 标签 ?
- 阴影：层级（或用边框 — 二选一，别同时上）

## 参考
- preview/ 情绪板图：[文件名]
- 外部参考：[链接 / 截图]

## 红线（继承 rules/design.md，按需覆盖）
- [本项目特例，如"主色用暖橙，覆盖 design.md 的禁蓝默认"]
```

> 此文件即 `rules/design.md` 优先级链里的 **`preview/`**——权威高于 design.md 默认规范。Builder 写 UI 前必须先读它。

## 异常路径

| 场景 | 处理方式 |
|------|----------|
| 四条路都试了还是没感觉 | 别硬逼。退回 design.md 白底简约默认先跑起来，看到真实界面再回头调（L2 阶段往往更容易拍板） |
| 图像 MCP 不可用 | 自动走 B/C，不阻塞、不报错 |
| 用户方向与 design.md 红线冲突 | design.md 已规定 `preview/` 优先——按用户方向走，提示一句即可，不要拦 |
| 中途想换方向 | 直接换路重来，已探索的偏好仍记入画像，不浪费 |
| 用户其实只想要默认 | 尊重，直接确认用 design.md 默认，不强行探索 |

## 使用示例

```
/taste                              # 给当前项目定设计方向
/taste 这是个面向年轻人的服饰电商后台
/taste 重新定调，现在的界面太死板
```

## 边界
- `/taste` 只定**方向**（产出 `preview/`），**不写 UI 代码**（那是 `/dev` 的活）
- 不替代 `rules/design.md`：design.md 是**没方向时的默认底线**，`/taste` 是**给出明确方向**
- 个人档案（L0）是可选增强，不建也能正常用
