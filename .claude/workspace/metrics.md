# 团队效能指标

> /dev 完成后自动追加记录。/standup 可查看趋势和分析。
> 用于定位团队瓶颈，持续改进配置。

---

## 指标定义

| 指标 | 含义 | 健康值 | 警告值 |
|------|------|--------|--------|
| spec 否决轮数 | 用户否决方案的次数 | 0-1 轮 | 3 轮 |
| /check 首次问题数 | 第一次快检发现的问题数 | 0-2 个 | 5+ 个 |
| /check 修复轮数 | 快检自动修复的轮数 | 1 轮 | 2 轮（上限） |
| /review-all 打回次数 | 全面审查不通过的次数 | 0 次 | 1+ 次 |
| 测试失败次数 | 测试不通过的次数 | 0 次 | 1+ 次 |
| 预估偏差 | 实际耗时 vs 预估 | < 50% | > 100% |

### 瓶颈分析

（基于数据自动判断）

- 如果 **spec 否决轮数高** → Architect-Planner 理解需求能力不足 → 建议：改进 Phase 0 需求确认
- 如果 **/check 首次问题数高** → Builder 代码质量不够 → 建议：加强 rules/ 或 TDD 流程
- 如果 **/review-all 打回频繁** → Builder 自检不足 → 建议：每个任务完成后强制 /check
- 如果 **测试失败频繁** → 整体质量把控不严 → 建议：加强测试规则
- 如果 **预估偏差大** → Architect 估算偏乐观 → 建议：调整估算系数
